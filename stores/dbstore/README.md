# Shared database Store

`dbstore` contains the single GORM implementation used by SQLite, MySQL, MariaDB, and PostgreSQL. It keeps checkpoint CAS, leases, fenced RunLog appends, retention, validation, and error semantics in one place while limiting dialect branches to schema setup, row locking, and concurrency-error classification.

Applications normally use one of the database-specific packages. SQLite, MySQL, MariaDB, and PostgreSQL expose the same `Migrate(dsn)` then `Open(dsn)` lifecycle: execute `Migrate` in the deployment stage, then let application instances call `Open`, which only connects and verifies the schema version and critical invariants. A true in-memory SQLite database is the sole exception because its schema cannot survive a separate connection. An existing v1 database requires an offline maintenance window: stop and drain every old writer, run `Migrate`, and start only v2-capable binaries after it succeeds. MySQL/MariaDB `GET_LOCK` and PostgreSQL advisory locking serialize migration processes but cannot stop an old application writer. Before recording v2, migration repairs and verifies every RunLog retention/registry row; rerunning `Migrate` is also the explicit repair path for incomplete v2 metadata.

All four database packages return their own `*Store` facade with the same persistence and retention methods. Advanced callers may use `SQLDB()` for connection-pool and health configuration or `GORMDB()` for application-specific GORM queries and transactions. Store still owns connection closure; callers must not mutate `gopact_*` tables or migration metadata, and Store methods do not join a transaction merely because it was opened through `GORMDB()`.

Encoded RunLog records and complete checkpoint records are capped at 4 MiB each. Terminal history remains append-only until the service invokes `PurgeTerminalRuns`; running and interrupted Runs are never selected. Purge first retires the Run atomically, then deletes history in row- and byte-bounded transactions while retaining a compact RunID tombstone that rejects late appends. `PurgeRunLog` is only for explicitly accepted replay loss and requires `AllowReplayLoss`; it never selects registry entries owned by this database's Workflow checkpointer. Its cutoff uses Store append time, not caller-supplied event time.

Long-running active Workflows can explicitly call `PurgeConfirmedRunLog` by RunID. It deletes only the contiguous prefix at or below the latest checkpoint's `ConfirmedSequence`, refuses checkpoints with a pending event, and persists a compaction floor so old sequence numbers cannot be reinserted. This preserves recovery from the current checkpoint, but old Retry/Fork targets, timeline projections, and audit records may disappear; callers must set `AllowHistoryLoss` and archive first when those features matter. A RunID query whose `After` cursor is below the floor returns `runlog.ErrHistoryCompacted` instead of silently returning an incomplete timeline.

The Store performs no background cleanup. Defaults are 1,000 logical history rows and 64 MiB of encoded record payload per purge call; a single oversized legacy row is deleted alone to guarantee progress. These budgets are not physical WAL/redo/binlog limits: indexes, retention metadata, MVCC/undo, page images, and replication retention add database-specific write amplification. Pace purge calls externally and monitor native database logs, replication slots/binlog consumers, backups, and vacuum/optimize health. Workflow tombstones grow with unique purged RunIDs, so high-cardinality deployments must include that compact table in capacity planning or archival/partition lifecycle design.

The Workflow runtime sends lease TTLs rather than trusting its wall-clock expiry. This Store derives and checks lease expiry from the database clock while holding the ownership lock, so host clock drift cannot create two database-valid owners. This still does not make arbitrary external side effects exactly once—use an API idempotency key or a business outbox/dedup record in the business database transaction.
